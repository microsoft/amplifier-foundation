# Bundle Documentation Convention (v2)

## Overview

The v2 bundle documentation system produces two complementary output types:

| Output | What it is | Location |
|--------|-----------|----------|
| `<name>.detail.md` | Composition table for ONE entry point | Co-located with source file |
| `overview.dot` / `overview.png` | Graph of ALL entry points + relationships | Repo root |

**Entry points** are the files that define bundle composition:
- `bundle.md` — root bundle
- `behaviors/*.yaml` — behavior bundles
- `bundles/*.yaml` — standalone bundles

---

## Per-Entry-Point Detail Files (`.detail.md`)

### What they contain

Each `.detail.md` is a structured markdown document with per-section tables showing:

- **Tools** — declared tool modules (schema token cost is `—`, injected at runtime)
- **Context Files** — `context.include:` entries with full file token cost estimates
- **Agents** — `agents.include:` entries with `meta.description` token cost estimates
- **Hooks** — hook definitions by name (~0 tokens)
- **Skills** — referenced skill files
- **Markdown body** — instruction cost for `.md` bundles (length ÷ 4)

A footer line contains the `source_hash` for freshness tracking.

### File naming

| Source file | Detail file |
|-------------|-------------|
| `bundle.md` | `bundle.detail.md` |
| `behaviors/my-capability.yaml` | `behaviors/my-capability.detail.md` |
| `bundles/code-review.yaml` | `bundles/code-review.detail.md` |

**Rule:** `<stem>.detail.md` — same stem, `.detail.md` extension, same directory.

### Token counting rules

| Element | Token estimate |
|---------|---------------|
| Context file | `len(file_content) // 4` |
| Agent description | `len(meta.description) // 4` |
| Markdown body | `len(body) // 4` |
| Tool schema | `—` (not countable from YAML) |
| Hook | ~0 |

### No LLM enhancement

Detail files are pure structured data generated from YAML/markdown parsing. They do **not** receive LLM label enhancement — the tables are already precise and machine-readable.

### Example footer (freshness tracking)

```
*Generated by dot_docs · source_hash: a3f2c1d8e9b047f2...*
```

## Overview DOT Graph (`overview.dot`)

### What it shows

The overview graph visualizes the entire repository at a glance:
- All entry points (root bundle, behaviors, standalone bundles)
- Include edges (which bundles compose which)
- Agent include edges
- A `source_hash` attribute for freshness tracking

### Location

`overview.dot` and `overview.png` are written to the **repo root**.

### Visual conventions

| Element | Shape | Fill color | Notes |
|---------|-------|------------|-------|
| Bundle (root) | `box` | `#4A90D9` (blue) | Root `bundle.md` |
| Included bundle | `box` | `#7EB8F7` (light blue) | `includes:` entries |
| Behavior | `component` | `#6AAF6A` (green) | `behaviors:` entries |
| Agent | `ellipse` | `#F5A623` (orange) | Agent `.md` files |
| Context file | `note` | `#F0E68C` (khaki) | `context.include:` entries |
| Tool/module | `cds` | `#B0B0B0` (grey) | `tools:` entries |
| Hook | `hexagon` | `#D9A0D9` (lavender) | `hooks:` entries |

### LLM enhancement

Overview DOT labels **are** LLM-enhanced by default: node labels are rewritten with concise, accessible English summaries rather than raw filenames. This makes the diagram readable by non-technical stakeholders.

To skip LLM enhancement (structural-only, faster):

```yaml
# In recipe context
enhance_diagrams: "false"
```

## Freshness Model

### Detail files

The `.detail.md` footer contains `source_hash: <sha256>` — the SHA-256 of the source file's raw content. When `validate-bundle-repo` checks freshness, it recomputes the hash from the current source and compares.

### Overview DOT

The `overview.dot` graph embeds `source_hash="<hash>"` as a graph attribute. The hash is derived from all entry point file contents combined. If any entry point changes, the hash changes and the overview is regenerated.

### Findings

| Code | Severity | Meaning |
|------|----------|---------|
| `BUNDLE_DOC_MISSING` | SUGGESTION | No `.detail.md` found co-located with source |
| `BUNDLE_DOC_STALE` | WARNING | `.detail.md` or `overview.dot` hash doesn't match current source |

## Generation

### Via validation (automatic)

`validate-bundle-repo` auto-regenerates stale or missing docs as a side effect of validation:

```
validate-bundle-repo → detects missing/stale → regenerates .detail.md + overview.dot
```

- Stale `.detail.md` files: regenerated immediately (no LLM)
- Stale `overview.dot`: regenerated with optional LLM label enhancement

### Direct (bulk generation)

```
generate-bundle-docs
```

Scans the repo for all entry points (`bundle.md`, `behaviors/*.yaml`, `bundles/*.yaml`) and regenerates their `.detail.md` files, plus `overview.dot` and `overview.png` at the repo root.

---

## Lifecycle: When to Regenerate

Regenerate when any entry point changes:

- `includes:` list (which bundles are composed)
- `behaviors:` list (which behaviors are added)
- `agents.include:` list
- `context.include:` list (files added/removed/renamed)
- `tools:` list (modules added/removed)
- `hooks:` definitions
- The description/frontmatter of the bundle itself
- Any agent `.md` file referenced in `agents.include:`

**Tip:** If unsure, run `validate-bundle-repo` — it checks hashes and regenerates only what's stale.

---

## Agent Documentation

Agent details (tools, context, model_role, description tokens) appear in the `.detail.md` of the **behavior or bundle that includes those agents** — not as separate per-agent DOT files.

This keeps agent documentation co-located with the composition context where agents are used, rather than scattered across individual agent files.

---

## Checklist

Before committing a bundle or behavior change:

- [ ] Co-located `.detail.md` exists alongside the source file
- [ ] `source_hash` in the `.detail.md` footer matches current source (or run `validate-bundle-repo`)
- [ ] `overview.dot` / `overview.png` exist at repo root and are fresh
- [ ] Token cost columns populated for context files and agents
